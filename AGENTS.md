## Deploy Workflow

**The VPS auto-deploys.** A cron job on the VPS (`/etc/cron.d/vdb-autodeploy`,
running `/usr/local/bin/vdb-autodeploy.sh` every minute) checks `origin/main` and
runs `git pull --ff-only` whenever the VPS is behind. After pulling, it runs
`python3 build.py`, which republishes `dist/` **in place** so the running nginx
container serves the new build with no restart. If the build fails, the previous
`dist/` keeps serving. **No manual VPS step needed.**

To ship a change, just commit to `main` and push:

```bash
git add -A && git commit -m "..."
git push          # auto-deploys to https://vdb-uk.com/ within ~1 minute
```

- **Auth:** the VPS pulls over SSH using a read-only deploy key (`/root/.ssh/vdb_deploy_key`,
  set via the repo's `core.sshCommand`) — no token stored in config.
- **Logs:** deploys are recorded in `/var/log/vdb-autodeploy.log` on the VPS;
  `/run/vdb-autodeploy.last` is a per-minute heartbeat.
- **The installed script must match `vdb-autodeploy.sh` in this repo.** If you
  change the repo copy, re-install it: `install -m755 vdb-autodeploy.sh
  /usr/local/bin/vdb-autodeploy.sh` (the two drifted apart once — see learnings).
- **Verify a deploy** with the `verify-deploy` skill (checks VPS HEAD, live HTTP,
  container health). **Manual deploy** (rarely needed): `source secrets.env` then
  `ssh root@$VPS_IP "cd /root/arbtechuk/vdb-uk-dot-com && git pull --ff-only && python3 build.py"`.
  `secrets.env` (gitignored) holds `VPS_IP` — prefer the Tailscale IP (100.64.222.60).

## Design System

- **Palette:** VDB brand — primary purple `#5E1A8E`, orange CTAs `#E89A3C`, teal secondary `#1C9C8E`, on the dark slate hero / stone-cream light backgrounds. NOTE: the rebrand (2026-06, see `docs/plans/30-06-2026-rebrand/`) remapped the token *values* in place — the `--red-*` custom properties now hold purple values, `--orange-500`/`--teal-500` are the accent tokens, and `--gold-*` is retained as a warm accent. So `var(--red-500)` renders purple. Don't be misled by the legacy token names.
- **Fonts:** Outfit (headings, 600) + DM Sans (body, 16px/1.65) — loaded from Google Fonts. (Was Crimson Pro serif before the 2026-06 rebrand.)
- **i18n:** Language/localisation runtime removed; English source is single-language.

## Conventions

- **Single-file source pages:** Keep page sources in `src/` (one file each) and share
  header/footer/scripts through `_partials/`, `site.js`, and `styles.css`.
  Includes use `<!-- @include _partials/header.html -->`. `dist/` is generated
  output and must not be edited directly.
- **Images:** Optimised JPGs under 200KB each in `images/`. Use descriptive filenames like `hero-secretariat.jpg`, `project-npk-field.jpg`.
- **Welsh translations:** Machine translation is acceptable for initial deployment but government-facing pages (especially Secretariat) need professional review before sharing with Welsh public sector contacts.
- **No frameworks:** Plain HTML/CSS/JS. `build.py` only. No npm. Keep it simple — Hannah's AI agent needs to be able to edit these files directly in `src/`.

## Engineering Principles

Write code and markup a non-expert could read and safely edit. Simplicity is the
brief here, not a constraint to work around.

- **Simplicity first.** No frameworks, no npm, no build step beyond `build.py`.
  If a change tempts you toward a toolchain, find the plain-HTML/CSS/JS way first.
- **Don't repeat yourself.** Markup shared across pages belongs in a `_partials/`
  include; repeated styling belongs in a named `styles.css` class — not copy-paste.
  If you paste the same block into a second page, stop and extract it.
- **Small and well-named.** Functions in `site.js`/`build.py` do one thing and say
  what they do. CSS classes are semantic and namespaced (e.g. `cs-hero__bg`).
- **Semantic, accessible HTML.** Correct heading order, `alt` text on images,
  labels/`aria` on controls, visible focus states. This is a public-sector-facing
  site — accessibility is a feature, not a nicety (see `accessibility.html`).
- **Refactor as you go (boy-scout rule).** When you touch a page and see it has
  drifted from the shared pattern (inlined a header, duplicated a component),
  reconcile it. Keep refactors surgical and reversible; don't fold a sweeping
  rewrite into a content change — separate commits.
- **Guard the invariants.** Preserve legal-entity strings (footer copyright,
  privacy data-controller), brand tokens (remember `--red-*` = purple), and the
  <200KB image budget. Never edit `dist/` by hand.
- **Verify before you ship.** Build (`build.py`) and preview locally (`preview`
  skill) before pushing; confirm it's live afterwards (`verify-deploy` skill).
  One focused change per commit with a clear message — `main` auto-deploys.

## Compounding Learning

The repo should get smarter every time we solve something, regardless of which
agent or session did it. That is what `docs/learnings/` is for.

- **Before non-trivial work,** skim `docs/learnings/` for a relevant prior lesson.
- **After solving something non-obvious** (a subtle bug, a gotcha, an infra
  quirk, a "why doesn't this work" that took real digging), write it down:
  1. Add a numbered file `docs/learnings/NNNN-short-title.md` using the template
     in `docs/learnings/README.md` (Problem / What we learned / How to apply).
  2. Add a one-line pointer to the index in `docs/learnings/README.md`.
  3. If the lesson implies a rule, also encode it where it will be enforced — a
     code comment at the point of risk, or a line in this file.
- Keep entries short and durable. One lesson per file. Link related plans/commits.

## Skills

Reusable workflows live in `.claude/skills/<name>/SKILL.md` (Claude Code
auto-discovers them as `/name`; Codex and other agents can read the same file as
a runbook). Each has YAML frontmatter (`name`, `description`, `user_invocable`,
optional `arguments`) followed by steps.

**Build a skill when a workflow recurs** — once you've done the same multi-step
task ~2–3 times (or a teammate would benefit from a repeatable, improvable
version), capture it. Improving the skill later compounds for everyone.

Current skills:
- `generate-image` — generate + optimise images via the Gemini API into `images/`.
- `preview` — build `dist/` and serve it locally for visual iteration.
- `verify-deploy` — confirm a push went live on the VPS (HEAD, live HTTP, container).
- `new-case-study` — scaffold a new case-study page from the shared template.
