## Human merge steps
1. Confirm you are on `refactor/dewelsh-components` and all desired commits are present.
2. Push branch changes for review if required by workflow.
3. Merge branch to `main` (or cherry-pick commits as policy requires).
4. Verify `main` tip has:
   - `_partials/head.html`
   - `_partials/header.html`
   - `_partials/footer.html`
   - `_partials/scripts.html`
   - `build.py`
   - `src/` full page set
   - `styles.css`
   - `site.js`
   - `dist/` in `.gitignore`
   - updated `docker-compose.yml`
   - updated `vdb-autodeploy.sh`
5. Confirm `lang-config.js` and all language-toggle scaffolding are removed.

## VPS cutover sequence
1. Deploy updated `main` (existing auto-pull schedule).
2. SSH to VPS and inspect `/usr/local/bin/vdb-autodeploy.sh` and container compose mount path.
3. Confirm `vdb-autodeploy.sh` now runs:
   - `git pull --ff-only`
   - `python3 build.py`
   - logs each attempt to `/var/log/vdb-autodeploy.log`
4. Validate `python3 --version` on the VPS host is available.
5. Watch `/var/log/vdb-autodeploy.log` for a successful pull + build cycle.
6. Confirm served site includes `dist` content (e.g., `/_/` request returns site shell and theme button exists).

## Rollback (post-merge)
- If any blocker appears after merge, execute the established rollback path for this repo:
  - revert the merge on `main`
  - push the revert
  - allow VPS to pull and rebuild.
- Keep this because `build.py` uses atomic temp swap; a bad VPS build should leave previous `dist/` in place.
